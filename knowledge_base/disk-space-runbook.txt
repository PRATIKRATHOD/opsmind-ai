# RUNBOOK: Production Linux Disk Space Full Troubleshooting
**Document Version:** 1.0  
**Target Audience:** SREs, Systems Administrators, DevOps  
**Objective:** Clear critical space blockages instantly on production filesystems without breaking running system integrity.

---

## 🚨 PHASE 1: Immediate Triage & Space Reclamation
When a disk fills up entirely (100% utilization), application writes fail, log streaming stops, and database engines drop connections to prevent corruption.

### 1.1 Check Global Filesystem Space Distribution
```bash
df -h
```
Identify the target mount point running out of storage capacity (e.g., `/`, `/var`, `/data`).

### 1.2 Locate the Largest Files & Aggressive Directories
Isolate exactly where the disk leak is happening by executing a depth search, sorting allocations by file size:
```bash
# Find top 15 largest directories starting from the root or target mount path
sudo du -ahx / 2>/dev/null | sort -rh | head -n 15
```

### 1.3 Safely Clear Massive Log Files
**CRITICAL WARNING:** Do not use `rm` to delete an active, open log file (e.g., `app.log`). If a running process keeps that file descriptor open, the disk space will **not** be released, even though the file name vanishes from directory trees.
Instead, safely truncate the file size to zero:
```bash
# Truncate the file securely without dropping active file handles
sudo truncate -s 0 /var/log/apps/offending_huge_file.log
# OR alternative shell standard redirect
sudo true > /var/log/apps/offending_huge_file.log
```

## 🔍 PHASE 2: Tracking Unreleased Open File Descriptors

### 2.1 Find Deleted Files Still Held in Memory by Processes
If `df -h` shows the disk is 100% full, but `du -sh` shows minimal file presence, files were deleted with `rm` while still being locked open by active processes:
```bash
sudo lsof +L1
# OR filter deleted files explicitly
sudo lsof | grep -i deleted
```
**Example output row:**
```text
java   12345  root   7w   REG   253,1  45G   0  /var/log/app.log (deleted)
```

### 2.2 Release the Trapped Space Without Downtime
To claim back that space without taking down a production application completely, safely clear the file allocations through the process descriptor pseudo-filesystem:
```bash
# Locate the Process ID (PID) and File Descriptor (FD) from the lsof output above
# Syntax: true > /proc/<PID>/fd/<FD_NUMBER>
sudo true > /proc/12345/fd/7
```
*The disk space will instantly reclaim back to the OS.*

## 🛠️ PHASE 3: Diagnosing Inode Exhaustion

### 3.1 Check Inode Capacity Limits
A disk can report "No space left on device" even if it has hundreds of gigabytes of physical space available. This happens when the filesystem runs completely out of Inodes (tracking metadata slots), caused by storing millions of tiny files:
```bash
df -i
```
If `IFree` reaches `0`, you are out of Inodes.

### 3.2 Isolate Directories Containing Excessive File Counts
```bash
# Locate the specific folders holding the largest quantity of individual files
sudo find / -xdev -type d -exec sh -c 'echo "$(find "$1" -type f | wc -l) $1"' _ {} \; | sort -rn | head -n 15
```
*Common culprits include old PHP session directories (`/var/lib/php/sessions`), micro-caching asset folders, or unmanaged application tmp files.*

## 🚀 PHASE 4: Permanent Architectural Hardening

### 4.1 Enforce Logrotate Rules
Ensure no application log directory operates without standard rotation parameters. Ensure files are aggressively compressed:
```text
# /etc/logrotate.d/app-logs
/var/log/apps/*.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    copytruncate
}
```
*Note: The `copytruncate` flag ensures logs are cleared inline without disrupting active system processes.*

### 4.2 System Cleaner Automated Cron Jobs
* [ ] Setup standard automated Docker/Podman image cleanup schedules:
  ```bash
  0 2 * * * /usr/bin/docker system prune -a --volumes -f
  ```
* [ ] Configure structured Prometheus/Grafana or CloudWatch storage monitoring alerts to breach when filesystems scale past `80%` capacity, providing a safe window for manual remediation.
